         If both SLAs breach, one P1 per breach is permitted.
         Contributing factors, secondary symptoms, and items that
         improve a non-breaching metric are P2 — even on a FAIL run.
         PASS runs: P1 is never assigned — no active breach exists.
    P2 — Addresses a contributing factor, secondary signal, or a
         finding that could cause a breach under higher load.
         Correct priority for all actionable findings on a PASS run.
         On FAIL runs: anything that does not directly fix the breaching
         metric is P2, regardless of perceived severity.
    P3 — Optimisation or observability improvement with no
         immediate impact on current metrics.

  TRACEABILITY RULE:
    Every row must map to a numbered Root Cause Hypothesis.
    Do not introduce causes or components not established above.
    The Bottleneck Analysis classification label (e.g. LATENCY-BOUND,
    THROUGHPUT-BOUND) must be cited verbatim — in all-caps — in at
    least one Recommendations row. Do not paraphrase it (e.g. do not
    write "latency-bound behaviour" — write "LATENCY-BOUND").

  IMPACT EXPRESSION:
    State impact in terms of the metrics already in this report.
    If estimable: "expected to reduce avg response time from
      X ms toward Y ms". If not estimable: "expected to eliminate
      HTTP 503 errors" or "expected to reduce p99 spike frequency".
    Do not write "will improve performance" without a metric anchor.
    P1 rows MUST additionally state the breaching SLA metric, its
    current observed value, and the threshold in the Expected Impact
    column — even when the action targets a mechanism (e.g. 504 errors).
    Correct: "Expected to reduce error rate on POST Submit Insurance
      Claim from 3.88% toward the 2.0% SLA threshold by eliminating
      504/503 upstream failures."
    Incorrect: "Expected to eliminate HTTP 504 and 503 errors."
      (mechanism only — missing SLA metric anchor)

  EFFORT DEFINITION:
    Low    — configuration change or parameter tuning; no code deploy.
    Medium — code change or infrastructure adjustment; single sprint.
    High   — architectural change; multiple sprints or quarters.

  Emit this table with the header row, populated from the findings:
  | Priority | Hypothesis | Action | Expected Impact | Effort |
  |----------|------------|--------|-----------------|--------|

  Minimum 1, maximum 7 rows.
  Write only rows traceable to a Root Cause Hypothesis with an actual signal.
  Do not fabricate rows to meet a minimum count.
  Exception — if no hypothesis has an actionable signal (e.g. a clean
  PASS run with no anomalous metrics), write instead:
    "No corrective actions are required — all observed metrics are within
    acceptable bounds."
  and omit the table entirely.
  Hypothesis column: write the hypothesis number (e.g. "#1", "#2")
    that this action addresses. Hypothesis numbers are globally sequential
    across the entire table — do not reset the counter when the priority
    level changes (e.g. P2 → P3). If the Root Cause Hypotheses section
    contains 4 hypotheses, valid values are #1, #2, #3, #4 only.
    A row referencing #1 after a row referencing #3 is a numbering error.
  Priority column: P1, P2, or P3 only — no other values.
  Effort column: Low, Medium, or High only — no other values.
  Sort rows by Priority ascending (P1 first), then Effort ascending
    within the same priority level.

  Audience: Everyone — QA engineer, delivery manager, SRE, and
            anyone deciding whether this build is production-ready.
            Write with zero ambiguity. No hedging, no conditionals,
            no qualifications. One clear outcome.

  DETERMINE THE VERDICT: read overallVerdictSummary.verdict from the input
  JSON. This is pre-computed by the plugin and MUST be used as-is — do NOT

  GUARD — only cite transaction labels that appear verbatim in
  errorSlaSummary, rtSlaSummary, or the "SLA Verdict" plain-text block
  from the input data. Do not introduce transaction names from general
