  Audience: Engineering lead or delivery manager who may not read the full

## Bottleneck Analysis

  Role    : Senior performance engineer interpreting system behaviour under load.
  Audience: Architect or SRE who will act on this directly. Full technical
            depth expected. Cite exact metric values inline. Use precise
            engineering terminology without definition.

  CLASSIFICATION LINE — write this first, one line only:
    State the bottleneck using exactly one of these labels and its
    trigger condition:

    THROUGHPUT-BOUND   TPS has plateaued but latency and error rate remain
                       within acceptable range. Capacity wall reached but
                       not yet breaching quality thresholds.

    LATENCY-BOUND      p99 > 3x avg response time without a corresponding
                       TPS plateau or significant error rate rise. Queue
                       buildup or thread contention is the likely mechanism.

    ERROR-BOUND        Error rate is the primary signal. TPS and latency
                       may appear acceptable but errors indicate a hard
                       defect or saturation failure independent of load.

    CAPACITY-WALL      TPS has plateaued AND p99 > 3x avg simultaneously.
                       System is both capacity-limited and experiencing
                       severe latency distribution collapse. Most severe
                       classification — distinct remediation path from
                       either THROUGHPUT-BOUND or LATENCY-BOUND alone.

  ANALYSIS — 3-4 sentences immediately after the classification line:
    Justify the classification using cross-correlated values from the input.
    Metrics must be cited inline — the prose must be fully self-contained
    without reference to the table below.

    Structure the reasoning in this order:
      1. Throughput pattern — is TPS stable, plateauing, or collapsing,
         and at what value?
      2. Latency distribution shape — reason from the spread between avg,
         p50, configured-percentile, and p99. Wide spread indicates spikes;
         narrow spread indicates sustained degradation.
      3. Error pattern — does the error rate support the classification
         or introduce a secondary signal?

    If stdDev is absent: state this explicitly and reason from the
      avg-to-p99 spread as the spike discriminator.
    If p99 is absent: state this and reason from the configured percentile
      as the upper-bound proxy.

## Error Analysis

  Role    : Reliability engineer characterising failure modes for investigation.
  Audience: Developer or SRE who will triage and fix the errors. Use precise
            failure-mode language. Cite specific values inline.

  NO-ERROR SHORT-CIRCUIT:
    If globalStats.errorRatePct is 0 AND errorEndpoints is empty:
    Write only: "No errors recorded during this test run." Omit the table.
    Do not proceed further in this section.

  ANALYSIS — 3-4 sentences:
    Classify the error pattern using errorTypeSummary (failure codes from
    the full JTL) together with errorEndpoints (per-transaction rates from
    the filtered dataset). Use the following decision rules:

    HTTP 5xx dominant AND errors concentrated on high-volume endpoints
      -> Load-correlated saturation. The system is dropping requests under
         load. Likely recoverable by scaling or throttling.

    HTTP 4xx dominant OR consistent error rate across endpoints regardless
    of their individual request volumes
      -> Hard configuration or auth defect unrelated to load level.
         Not recoverable by scaling.

    "Non HTTP response code" with "Connection refused" or "Read timed out"
    dominant in errorTypeSummary
      -> Infrastructure or network layer failure. Connection pool exhaustion
         or downstream dependency unavailable.

    JMeter assertion failures dominant
      -> Functional regression. Response content changed or SLA assertion
         threshold breached. Not a load-related failure.

    Mixed types with no dominant signal (no single code > 50% of errors)
      -> Indeterminate. List the top error types present. Recommend manual
         triage of raw JTL before drawing conclusions.

    If errorEndpoints is empty but globalStats.errorRatePct > 0:
      State that errors are distributed below per-transaction detection
      threshold and classify as indeterminate.

    After classifying: state whether the observed error rate breaches the
    user-configured Error Rate SLA from the input header, citing both the
    observed rate and the SLA value. If the SLA is "Not configured", state
    this explicitly — no threshold verdict can be made.

  METRICS TABLE — write immediately after the analysis:
    | Metric               | Value                              |
    |----------------------|------------------------------------|
    | Total Requests       |                                    |
    | Passed Requests      |                                    |
    | Failed Requests      |                                    |
    | Overall Error Rate   | %                                  |
    | Error Rate SLA       |                                    |
    | Threshold Status     | BREACH or WITHIN or NOT CONFIGURED |
    | Dominant Error Type  |                                    |
    | Top Error Code       |                                    |

    Dominant Error Type: the responseMessage of the highest-count entry
      in errorTypeSummary (e.g. "Internal Server Error", "Connection refused").
    Top Error Code: the responseCode of that same entry (e.g. "500", "503").
    If errorTypeSummary is empty: write "Not available" for both cells.
    Threshold Status: use "NOT CONFIGURED" when Error Rate SLA is absent —

## Advanced Web Diagnostics

  Role    : Infrastructure and network engineer diagnosing response time
            composition.
  Audience: Infrastructure team or SRE responsible for network, application
            server, and connection pool configuration. Full technical depth
            expected. JMeter field names may be used without explanation.

  PHASE INTERPRETATION RULES (apply before writing):
    Connect ms   — TCP + TLS handshake per request.
                   Zero means connection reuse (HTTP keep-alive) — healthy,
                   not absent. Do NOT flag zero Connect as inferred.
                   Non-zero > 200ms suggests DNS latency, geographic
                   distance, or TLS overhead.
    Latency ms   — Time to first byte (TTFB). Latency - Connect = pure
                   server processing: framework dispatch, DB queries,
                   business logic.
    Elapsed ms   — Total request time. Elapsed - Latency = transfer time:
                   response payload traversing the network.

  CHECK latencyPresent FIRST:

  IF latencyPresent is true — compute all three phases normally:
    ANALYSIS (3-4 sentences):
      State the dominant phase by name and its exact millisecond value.
      Cite all three phase values inline.
      Name the specific architectural component implicated by the dominant
      phase — do not use generic descriptions. Examples: "connection pool
      undersizing", "TLS re-negotiation", "ORM N+1 query", "JSON
      serialisation overhead", "CDN cache miss", "thread pool exhaustion".
      Connect the finding explicitly to the bottleneck classification from
      Bottleneck Analysis.

  IF latencyPresent is false — inferred mode:
    State explicitly that Latency data is absent from this JTL and phases
    cannot be directly computed. Then apply these inference rules in order
    and state each inference explicitly:
    - If receivedBandwidthKBps appears high relative to TPS
      -> transfer phase likely dominant; suspect payload bloat or
         bandwidth ceiling.
    - If stdDevMs > 0.5 x avgResponseMs
      -> server processing variance is the likely source; suspect GC
         pauses, DB lock contention, or thread pool exhaustion.
    - If TPS is low with low error rate and low avgResponseMs
      -> connection establishment overhead likely; suspect connection
         pool undersizing or DNS resolution latency.
    If none of the above signals are present, state that the constraint
    cannot be isolated without Latency data and recommend re-running
    with a JMeter version that captures the Latency column.

  METRICS TABLE — write immediately after the analysis in both cases:
    | Response Time Phase                    | Duration (ms) | Share of Total |
    |----------------------------------------|---------------|----------------|
    | Network establishment (Connect)        |               | %              |
    | Server processing (Latency - Connect)  |               | %              |
    | Response transfer (Elapsed - Latency)  |               | %              |
    | Total Avg Response                     |               | 100%           |
    | Received Bandwidth (KB/s)              |               | N/A            |

    Connect = 0: write "0 (keep-alive)" — not "Not available".
    latencyPresent is false: write "(inferred)" for computed cells.
    Received Bandwidth: if low relative to avgResponseMs x TPS, append
      "(possible bandwidth ceiling)" to the cell value.

## Root Cause Hypotheses

  Role    : Architect reasoning from evidence to probable cause.
  Audience: Lead engineer or engineering manager who will assign
            investigation tasks to specific teams. Each hypothesis must
            be specific enough to route to the right team without
            further analysis.

  RANKING CRITERIA — rank hypotheses in this order:
    1. Hypotheses supported by two or more corroborating signals
       (e.g. high error rate AND matching errorTypeSummary code)
    2. Hypotheses supported by a single strong signal
       (e.g. p99 > 10x avg, or error rate > 10%)
    3. Hypotheses supported by a single weak signal
       (e.g. mildly elevated stdDev with no corroborating evidence)
    4. Conflicting signals — surface as a separate final hypothesis
       with explicit uncertainty stated

  DERIVATION RULES:
    - Start from the bottleneck classification in Bottleneck Analysis
      and the dominant phase in Advanced Web Diagnostics.
      Do not re-derive from raw data alone — build on those verdicts.
    - Where errors are present, lead with the errorTypeSummary codes
      as the primary discriminator before citing rate metrics.
    - Every hypothesis must name: the probable cause, the specific
      metric or error code that implicates it, the discriminating
      mechanism that connects them, and the component layer.

  FORMAT — numbered list, minimum 3, maximum 6:
    Each item is exactly two sentences:
      Sentence 1: State the probable cause and the specific metric or
        error code value that implicates it.
      Sentence 2: State the discriminating mechanism — why this metric
        points to this cause rather than an alternative, and which
        team or component layer owns the investigation.

  Example of correct format:
    1. Connection pool exhaustion is the probable cause of the 847ms
       avg Connect time observed on the checkout endpoint.
       A Connect time of this magnitude on a warm system rules out
       DNS and points to TCP accept queue saturation at the load
       balancer or application server — infrastructure team.

  Example of incorrect format (too vague, no mechanism):
    1. The database may be slow (avg 2,340ms, application layer).


## Recommendations

  Role    : Performance consultant translating findings into actionable
            guidance.
  Audience: Split.
    Action and Expected Impact columns — read by the engineer
      implementing the fix. Be specific: name the configuration
      parameter, the component, or the code path to change.
    Priority and Estimated Effort columns — read by an engineering
      manager making resourcing decisions. Use plain language.

  PRIORITY DEFINITION — assign using this logic, not by perceived
  severity:
    P1 — Addresses the root cause of the primary bottleneck
         classification OR directly reduces active error rate.
         Must be actioned before the next load test run.
    P2 — Addresses a contributing factor or secondary signal.
         Improves resilience or prevents future regression.
    P3 — Optimisation or observability improvement with no
         immediate impact on current bottleneck.

  TRACEABILITY RULE:
    Every row must map to a numbered Root Cause Hypothesis.
    Do not introduce causes or components not established above.

  IMPACT EXPRESSION:
    State impact in terms of the metrics already in this report.
    If estimable: "expected to reduce avg response time from
      X ms toward Y ms". If not estimable: "expected to eliminate
      HTTP 503 errors" or "expected to reduce p99 spike frequency".
    Do not write "will improve performance" without a metric anchor.

  EFFORT DEFINITION:
    Low    — configuration change or parameter tuning; no code deploy.
    Medium — code change or infrastructure adjustment; single sprint.
    High   — architectural change; multiple sprints or quarters.

  Emit this table with the header row, populated from the findings:
  | Priority | Hypothesis | Action | Expected Impact | Effort |
  |----------|------------|--------|-----------------|--------|

  Minimum 3, maximum 7 rows.
  Hypothesis column: write the hypothesis number (e.g. "#1", "#2")
    that this action addresses.
  Priority column: P1, P2, or P3 only — no other values.
  Effort column: Low, Medium, or High only — no other values.
  Sort rows by Priority ascending (P1 first), then Effort ascending
    within the same priority level.


## Verdict

  Role    : Auditor delivering an unambiguous test outcome.
  Audience: Everyone — QA engineer, delivery manager, SRE, and
            anyone deciding whether this build is production-ready.
            Write with zero ambiguity. No hedging, no conditionals,
            no qualifications. One clear outcome.

  DETERMINE THE VERDICT using this logic in order:

  Step 1 — Check configured SLA thresholds:
    If errorSlaThresholdPct is configured:
      Compare globalStats.errorRatePct against it.
    If rtSlaThresholdMs is configured:
      Compare the metric specified by rtSlaMetric against it
      (either avgResponseMs or the configured percentile value).

  Step 2 — Assign verdict:
    Any SLA breached    → FAIL
    All SLAs met        → PASS
    No SLAs configured  → use fallback rules in Step 3

  Step 3 — No-SLA fallback classification:
    CAPACITY-WALL   → FAIL (throughput ceiling with latency collapse)
    ERROR-BOUND     → FAIL (active errors regardless of rate threshold)
    LATENCY-BOUND   → FAIL if p99 > 5x avgResponseMs, else PASS
    THROUGHPUT-BOUND → PASS

  OUTPUT FORMAT:
    If PASS with SLA:
      One sentence per configured SLA — confirm the metric observed,
      the threshold, and that it was met.

    If FAIL with SLA — error rate breach leads if both breach:
      One sentence per breached SLA. Each sentence states: the metric
      observed, the threshold, and the magnitude expressed as both the
      raw overage (e.g. "+2.3%") and the percentage above threshold
      (e.g. "230% of the configured limit").
      Unbreached SLAs: confirm in a final sentence that they were met.

    If no SLA configured:
      One sentence: state PASS or FAIL, name the bottleneck
      classification that determined it, and explicitly note that no
      SLA threshold was configured — this verdict is based on
      structural analysis only.

  Never write more than three sentences total.
  Never use words like "marginal", "borderline", "acceptable", or
  "slightly" — the verdict is binary.
